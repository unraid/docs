# Repository XML Format

Your submission repository must contain valid XML templates for each Docker application or plugin you want listed in Community Applications. Each app entry needs either a Docker image repository or a plugin manifest URL so the submission pipeline can identify what is being published.

If you use the [starter repository](https://github.com/unraid/unraid-community-apps-starter), remove example files you do not need and replace every placeholder before submitting.

## Docker app templates

Use one XML file per Docker app template. At minimum, include a display name, image repository, support/project links, a short overview, and a category.

```xml
<Container version="2">
  <Name>Example App</Name>
  <Repository>ghcr.io/example/example-app:latest</Repository>
  <Registry>https://ghcr.io</Registry>
  <Network>bridge</Network>
  <Shell>bash</Shell>
  <Privileged>false</Privileged>
  <Support>https://forums.unraid.net/topic/example-support-thread</Support>
  <Project>https://github.com/example/example-app</Project>
  <Overview>Short summary shown in Community Applications.</Overview>
  <Description>Longer description shown in Community Applications.</Description>
  <Category>Tools:Utilities</Category>
  <WebUI>http://[IP]:[PORT:8080]</WebUI>
  <TemplateURL>https://raw.githubusercontent.com/example/community-apps/main/templates/example-app.xml</TemplateURL>
</Container>
```

### Docker template checklist

- Use `<Repository>` for the Docker image reference.
- Use `<TemplateURL>` for the canonical raw URL of the template file.
- Use `<Overview>` for the primary summary users and moderators will read.
- Use `<Support>` and `<Project>` links that point to real destinations.
- Add `<Config>` entries for paths, ports, variables, devices, or labels that users should configure through the Unraid template UI.

## Plugin templates

Plugin wrapper XML files identify Unraid plugin submissions and point to the plugin manifest.

```xml
<Plugin>
  <Name>Example Plugin</Name>
  <PluginURL>https://raw.githubusercontent.com/example/example-plugin/main/example-plugin.plg</PluginURL>
  <Support>https://forums.unraid.net/topic/example-support-thread</Support>
  <Project>https://github.com/example/example-plugin</Project>
  <Overview>Explain what this plugin does and why users should install it.</Overview>
  <Category>Tools:Utilities</Category>
</Plugin>
```

### Plugin template checklist

- Use `<PluginURL>` for the raw `.plg` manifest URL.
- Include support and project links so moderators can confirm provenance and users know where to get help.
- Keep the overview specific to what the plugin changes in Unraid OS.
- Expect manual review for plugin submissions.

## Validation

Run validation and scan checks in the [submission portal](https://ca.unraid.net/submit) after every meaningful XML change. Fix scanner errors in the repository, commit them, and scan again before submitting for review.
