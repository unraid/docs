# Version 7.3.0 2026-05-12

Dieses stabile Release ergänzt interne Boot-Unterstützung, einen Onboarding-Assistenten, TPM-basierte Lizenzierung, Verbesserungen beim Docker-Netzwerk, erweiterte Hardwareunterstützung, ZFS- und Storage-Fixes, Aktualisierungen der Virtualisierung, eine Sicherheitshärtung der WebGUI und Paketaktualisierungen.

## Upgrade

Für schrittweise Anweisungen siehe [Unraid aktualisieren](/unraid-os/system-administration/maintain-and-update/upgrading-unraid/). Fragen zu Ihrer [Lizenz](/unraid-os/troubleshooting/licensing-faq/)?

### Bekannte Probleme {#known-issues}

- Während der Einrichtung des internen Boots kann die WebGUI während der Konfiguration des internen Boots **Array Offline** anzeigen. Starten Sie Ihr Flash-Laufwerk während dieses Vorgangs **NICHT** manuell neu und entfernen Sie es nicht.
- Interner Boot erfordert Boot-Speicher, auf den während des Starts von integrierten Linux-Treibern zugegriffen werden kann. Geräte, die Drittanbietertreiber oder eine Konfiguration nach dem Start erfordern, können für internen Boot nicht verwendet werden. Unraid unterstützt die folgenden Treiber für internen Boot:
  - ahci
  - mmc
  - mpt3sas
  - mvsas
  - nvme
  - smartpqi
  - USB (`usb-storage` und `uas`, auch bekannt als UASP)
- Wenn ein Boot-Pool mit zwei Geräten ein Gerät mit einer unformatierten Datenpartition enthält, wird das Entfernen eines Boot-Pool-Geräts möglicherweise nicht korrekt abgeschlossen.
- Wenn ein geteiltes Boot-Pool-Gerät entfernt und unmittelbar danach ein anderes Gerät hinzugefügt wird, wird die Slot-Anzahl möglicherweise nicht immer aktualisiert, bis der Server neu gestartet wird.
- Einige USV-Geräte melden möglicherweise **Online No battery detected** und protokollieren wiederholt Meldungen vom Typ `Event data for report 22 was too short`. Falls betroffen, kann das NUT-Plugin einen funktionierenden Überwachungspfad für die USV bereitstellen.
- Einige Docker-Container in VLAN-Netzwerken erhalten nach dem Upgrade von 7.2.4 oder 7.2.5 möglicherweise keine DHCP-Zuweisung oder verwenden nicht die erwartete statische Netzwerkkonfiguration. Dieses Release enthält eine mögliche Abhilfe für dieses Problem.

### Anmerkungen

- Das Hinzufügen, Entfernen und Ersetzen von Geräten im ZFS-Boot-Pool funktioniert korrekt. Wenn Sie mit einem dedizierten Boot-Pool aus zwei kleinen Geräten beginnen und diese nacheinander durch größere Geräte ersetzen, kann der Pool in einen geteilten Pool umgewandelt werden.
- Wenn dem Boot-Pool Geräte hinzugefügt, daraus entfernt oder darin ersetzt werden, aktualisiert Unraid derzeit die BIOS-Boot-Priorität des Servers nicht automatisch. Überprüfen Sie nach diesen Änderungen die BIOS-Boot-Reihenfolge manuell.

### Rückschritte {#rolling-back}

- Gehen Sie nicht auf `7.2.6` oder früher zurück, nachdem Sie den internen Boot oder die TPM-basierte Lizenzierung aktiviert haben.
- Wenn Sie ZFS-Pool-Features aktualisieren, können frühere Unraid-Versionen diese Pools möglicherweise nicht importieren.
- Wenn Sie ein Rollback durchführen müssen, folgen Sie dem normalen Rollback-Prozess, bevor Sie zwischen Builds wechseln.

Wenn Sie einen Rückweg auf eine Version unterhalb von `7.3.0` in Betracht ziehen, lesen Sie auch die [Versionshinweise zu 7.2.6](/unraid-os/release-notes/7.2.6/).

## Änderungen vs. [7.2.6](/unraid-os/release-notes/7.2.6/)

### Sicherheit

- Sicherheit: WebGUI-Sicherheits-Härtung für die Verarbeitung von Anfragen authentifizierter Sitzungen hinzufügen. Dieser Fix ist bereits in 7.2.5 enthalten.
- Sicherheit: Pakete aktualisieren
- Sicherheitsnotiz: Die enthaltenen Kernel-Patches 7.3 beheben zwei lokale Schwachstellen zur Privilegieneskalation: Copy Fail (CVE-2026-31431) und Dirty Frag (CVE-2026-43284 und CVE-2026-43500)

### Onboarding und interner Boot

Neue Benutzer starten automatisch im Onboarding-Assistenten, wo sie Sprache, Zeitzone, Design, Aktivierungscode-Einstellungen und die Boot-Option auswählen, einschließlich internem Boot.

Bestehende Benutzer können zu **_Tools → Onboarding-Wizard_** gehen, um die Einstellungen zu überprüfen und zu internem Boot zu migrieren.

Details finden Sie unter [Onboarding](/unraid-os/getting-started/set-up-unraid/deploy-and-configure-unraid-os/).

### Lizenzierung

Alle neuen und Ersatzschlüssel verwenden nach Möglichkeit TPM-basierte Lizenzierung, weil sie zuverlässiger ist als Flash-basierte Lizenzierung.

Informationen zum proaktiven Wechsel zur TPM-basierten Lizenzierung finden Sie unter [Wie kann ich von Flash-basierter Lizenzierung zu TPM-basierter Lizenzierung wechseln](/unraid-os/troubleshooting/tpm-licensing-faq/#tpm).

### Container / Docker

- Verbesserung: Docker von `29.3.1` auf `29.4.3` aktualisieren.
- Neu: Ein optionales festes MAC-Adressen-Feld zu Docker-Vorlagen für Container hinzufügen, die über Neustarts hinweg eine stabile Netzwerkidentität benötigen, einschließlich DHCP-Reservierungen, Router- oder Firewall-Regeln, Switch-ACLs und Monitoring.
- Verbesserung: MAC-Adressen von Docker-Containern in der erweiterten Ansicht zusammen mit den vorhandenen Netzwerk- und IP-Informationen anzeigen.
- Fehlerbehebung: Alte Werte `--mac-address=` aus den Zusätzlichen Parametern nach Möglichkeit in das neue Feld für feste MAC-Adressen migrieren, während Vorlagen unverändert bleiben, wenn das Networking weiterhin von den Zusätzlichen Parametern gesteuert wird.
- Fehlerbehebung: Die fehlende Unterstützung hinzufügen, die für Strix Point iGPU-Docker-Workloads auf betroffenen Systemen erforderlich ist. Siehe den [Benutzerbericht](https://product.unraid.net/p/strix-point-igpu-support).
- Fix: Die WebGUI reaktionsfähig halten, während Docker-Container über die Seite "Container hinzufügen" hinzugefügt werden.

### Speicher

- Neu: Beschädigte Dateien im ZFS-Pool-Status anzeigen.
- Neu: Eine benutzerseitige Steuerung für die maximale ZFS-ARC-Größe unter **_Einstellungen → Festplatteneinstellungen → Tunable (zfs_arc_max)_** hinzufügen. Dies wurde bisher über einen benutzerdefinierten ZFS-Treiberparameter gesteuert. Möglicherweise müssen Sie Ihre Einstellung an den neuen Speicherort migrieren.
- Verbesserung: Sichtbarkeit von Pool- und Laufwerkszustandsinformationen verbessert.
- Verbesserung: Erkennung doppelter Laufwerke und zugehörige Meldungen verbessert.
- Fehlerbehebung: Verhindert, dass Laufwerke während des Paritätskopierens als Teil des Paritätstauschs herunterfahren.
- Fehlerbehebung: Korrigiert, dass ZFS-Pools in betroffenen Fällen einmal alle 24 Stunden aufwachen.
- Fix: Die Behandlung des ZFS-Pool-Austauschs korrigieren, wenn ein entferntes Gerät als `REMOVED` markiert ist, sodass Austauschabläufe den Pool nicht mehr degradiert mit einem veralteten blauen Zustand des neuen Geräts oder wiederholter Überschreibungswarnung zurücklassen.
- Fehlerbehebung: Verhindert die Erstellung von ZFS-Pools, deren Namen reservierte ZFS-Namenspräfixe wie `mirror`, `raidz`, `draid` oder `spare` verwenden.

- Fehlerbehebung: Korrigiert, dass das Bereinigen nach dem Commit eines VM-Snapshots die Snapshot-Metadaten nicht korrekt aktualisiert.

- Fix: Veraltete VM-Maschinentypen beim Start aktualisieren, damit ältere VMs nach dem Upgrade nicht mit nicht unterstützten QEMU-Maschinentyp-Fehlern ausfallen.
- Fix: Die zugewiesene `RenderGPU` für die erste VM in der VM-Liste anzeigen, wenn Virtio 3D verwendet wird.
