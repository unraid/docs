# Version 7.3.0 2026-05-12

Cette version stable ajoute la prise en charge du démarrage interne, un assistant d'intégration, une licence basée sur TPM, des améliorations du réseau Docker, une prise en charge matérielle étendue, des corrections ZFS et de stockage, des mises à jour de la virtualisation, un durcissement de la sécurité de WebGUI et des mises à jour de paquets.

## Mise à niveau

Pour des instructions étape par étape, consultez [Mise à jour d'Unraid](/unraid-os/system-administration/maintain-and-update/upgrading-unraid/). Des questions sur votre [licence](/unraid-os/troubleshooting/licensing-faq/) ?

### Problèmes connus {/* #known-issues */}

- Pendant la configuration du démarrage interne, WebGUI peut afficher **Array Offline** pendant la configuration du démarrage interne. **NE REDÉMARREZ PAS** manuellement et ne retirez pas votre clé USB pendant ce processus.
- Le démarrage interne nécessite un support de démarrage accessible au démarrage par les pilotes Linux intégrés au noyau. Les périphériques qui nécessitent des pilotes tiers ou une configuration après le démarrage ne peuvent pas être utilisés pour le démarrage interne. Unraid prend en charge les pilotes suivants pour le démarrage interne :
  - ahci
  - mmc
  - mpt3sas
  - mvsas
  - nvme
  - smartpqi
  - USB (`usb-storage` et `uas`, également connus sous le nom de UASP)
- Si un pool de démarrage à deux périphériques contient un périphérique avec une partition de données non formatée, la suppression d'un périphérique du pool de démarrage peut ne pas se terminer correctement.
- Si un périphérique de pool de démarrage fractionné est retiré puis qu’un autre périphérique est ajouté immédiatement après, le nombre d’emplacements peut ne pas toujours se mettre à jour tant que le serveur n’a pas été redémarré.
- Certains périphériques UPS peuvent signaler **En ligne Aucune batterie détectée** et consigner à plusieurs reprises les messages `Event data for report 22 was too short`. Si cela vous affecte, le plugin NUT peut fournir un chemin de surveillance UPS fonctionnel.
- Certains conteneurs Docker sur des réseaux VLAN peuvent ne pas recevoir DHCP ou utiliser la configuration réseau statique attendue après une mise à niveau depuis la version 7.2.4 ou 7.2.5. Cette version propose une atténuation possible de ce problème.

### Notes

- L'ajout, la suppression et le remplacement de périphériques dans le pool de démarrage ZFS fonctionnent correctement. Si vous commencez avec un pool de démarrage dédié composé de deux petits périphériques et que vous les remplacez un par un par des périphériques plus grands, le pool peut se convertir en pool fractionné.
- Lorsque des périphériques du pool de démarrage sont ajoutés, supprimés ou remplacés, Unraid ne met actuellement pas à jour automatiquement la priorité de démarrage du BIOS du serveur. Après avoir effectué ces modifications, vérifiez manuellement l'ordre de démarrage du BIOS.

### Rétrogradation {/* #rolling-back */}

- Ne revenez pas à `7.2.6` ou à une version antérieure après avoir activé le démarrage interne ou la licence basée sur le TPM.
- Si vous mettez à niveau les fonctionnalités des pools ZFS, les versions antérieures de Unraid pourraient ne pas être en mesure d’importer ces pools.
- Si vous devez revenir en arrière, suivez le processus normal de retour arrière avant de passer d'une build à une autre.

Si vous envisagez un retour à une version antérieure à `7.3.0`, consultez également les [notes de version de la 7.2.6](/unraid-os/release-notes/7.2.6/).

## Modifications par rapport à la [version 7.2.6](/unraid-os/release-notes/7.2.6/)

### Sécurité

- Sécurité : ajouter un renforcement de la sécurité de WebGUI pour la gestion des requêtes de session authentifiée. Ce correctif est déjà inclus dans 7.2.5.
- Sécurité : mise à jour des paquets
- Note de sécurité : les correctifs du noyau 7.3 inclus corrigent deux vulnérabilités locales d’élévation de privilèges : Copy Fail (CVE-2026-31431) et Dirty Frag (CVE-2026-43284 et CVE-2026-43500)

### Intégration et démarrage interne

Les nouveaux utilisateurs démarrent automatiquement dans l’assistant d’intégration, où ils choisissent la langue, le fuseau horaire, le thème, les paramètres du code d’activation et l’option de démarrage, y compris le démarrage interne.

Les utilisateurs existants peuvent aller dans **_Tools → Onboarding Wizard_** pour vérifier les paramètres et migrer vers le démarrage interne.

Pour plus de détails, consultez [l’intégration](/unraid-os/getting-started/set-up-unraid/deploy-and-configure-unraid-os/).

### Licence

Toutes les nouvelles clés et les clés de remplacement utilisent une licence basée sur le TPM lorsque cela est possible, car elle est plus fiable qu’une licence basée sur le flash.

Pour passer de manière proactive à une licence basée sur le TPM, consultez [Comment passer d’une licence basée sur le flash à une licence basée sur le TPM](/unraid-os/troubleshooting/tpm-licensing-faq/#tpm).

- Amélioration : mettre à jour Docker de `29.3.1` à `29.4.3`.

- Amélioration : afficher les adresses MAC des conteneurs Docker dans la vue avancée, à côté des informations réseau et IP existantes.
- Correction : migrer les anciennes valeurs `--mac-address=` depuis les paramètres supplémentaires vers le nouveau champ d’adresse MAC fixe lorsque cela est sûr, tout en laissant les modèles inchangés lorsque la mise en réseau dépend encore des paramètres supplémentaires.
- Correction : ajouter la prise en charge manquante nécessaire aux charges de travail Docker sur iGPU Strix Point sur les systèmes concernés. Voir le [rapport utilisateur](https://product.unraid.net/p/strix-point-igpu-support).
