## 作成ルール

- ユーザーへの影響を起点に記述してください。エンジニアリングのメモは、ユーザー、管理者、またはアップグレードの安全性に対して何が変わったのかに言い換えてください。
- 箇条書きは簡潔かつ事実ベースにしてください。ユーザーにとって重要でない限り、社内のプロジェクト名、課題 ID、ブランチ名、コミットハッシュ、実装詳細は含めないでください。
- 変更セクションでは、`New:`、`Improvement:`、`Fix:`、`Security:` の接頭辞を一貫して使用してください。
- 関連する項目は、最も具体的な見出しの下にまとめてください。広範な UI、設定、システムサービス、ライセンス状態、通知の変更には `WebGUI / System` を使用してください。
- `BREAKING CHANGES` セクションは、ユーザーが動作を変更する必要がある場合、またはロールバック、互換性、ネットワーク、ストレージ、データ、設定の動作が変わる場合にのみ含めてください。
- CVE、影響を受けるパッケージ、パッケージのバージョン、ロールバック警告、既知の問題、またはアップグレード推奨事項を作り出さないでください。ソース資料が不明確な場合は、`[NEEDS CONFIRMATION: ...]` と記載してください。
- 公開されているソースリンクは、上流のリリースノート、ユーザー報告、関連ドキュメントなど、詳細の検証に役立つ場合は保持してください。
- ドキュメントリンクは、次の形式を優先してください: `/unraid-os/release-notes/[VERSION]/`。
- UI ラベルには **太字** を、***Settings → Disk Settings*** のようなナビゲーションパスには ***太字斜体*** を使用してください。
- パッケージ名、コマンド、設定キー、パス、バージョン、カーネル設定シンボル、エラー文字列にはインラインコードを使用してください。
- 公開前に、未使用の任意セクションを削除してください。

## リリースノートのチェックリスト

公開前に、次の点を確認してください:

- 見出しは `# Version [VERSION] [YYYY-MM-DD]` です。
- 概要には、各セクションを繰り返さずに最も重要な変更が記載されています。
- アップグレード、既知の問題、およびロールバックに関する注記は、このリリースについて正確です。
- セキュリティに関する記述は、確認済みのアドバイザリまたはパッケージの変更履歴と一致しています。
- パッケージ数はパッケージ一覧と一致しています。
- 前回リリースへのリンクは正しいバージョンを指しています。
- プレースホルダー付きの任意セクションは削除されています。
